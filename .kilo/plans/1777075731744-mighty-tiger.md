# Plan: Fix Local Review Findings

## Goal
Address the four findings from the local review of `codex/apply-openspec-change` without broad refactoring:

1. Recreate the no-preset warning window after it is closed.
2. Make `SliderProfileJsonService` independent of `JsonDocument` lifetime assumptions.
3. Stop `AsyncRelayCommand` from always passing `CancellationToken.None`.
4. Guard the discard-confirmation dialog cancellation callback against closing an already-closed window.

## Implementation Steps

### 1. Fix No-Preset Window Reuse
In `WindowNoPresetNotificationService.EnsureWindow()`:

- After creating `window`, capture it in `createdWindow`.
- Register a `Closed` handler mirroring `WindowImageViewService`:
  - Ignore stale close events if `window` no longer references the same instance.
  - Set `customTargetsList = null`.
  - Set `npcsList = null`.
  - Set `window = null`.

Add a focused Avalonia headless test similar to `WindowImageViewServiceTests.ShowImageRecreatesWindowAfterClose`:

- Call `ShowTargetsWithoutPresets(...)` with one empty target.
- Reflect the private `window` field and assert it is visible.
- Close it.
- Assert the private `window` field is null.
- Call `ShowTargetsWithoutPresets(...)` again.
- Assert a new visible window is created and is not the same instance.
- Close the second window.

### 2. Materialize Slider Profile JSON Sequences
In `SliderProfileJsonService.LoadFromString()`:

- Keep `using var document = JsonDocument.Parse(json);`.
- Pass arrays to the constructor:
  - `ReadDefaults(root).ToArray()`
  - `ReadMultipliers(root).ToArray()`
  - `ReadInverted(root).ToArray()`

Testing:

- Existing template generation tests should continue to cover profile loading behavior.
- Add a tiny unit test if desired to load a profile string and assert defaults/multipliers/inverted behavior still works after `LoadFromString()` returns. This is optional because the production regression is mostly a maintainability/lifetime hardening issue, not an observable current failure.

### 3. Make AsyncRelayCommand Use a Real Token
In `AsyncRelayCommand`:

- Add a private `CancellationTokenSource?` field for the active execution.
- In `Execute`:
  - Return early when `CanExecute(parameter)` is false.
  - Create a new `CancellationTokenSource` for the invocation.
  - Cancel and dispose any prior active CTS to make re-entry cancel the previous invocation.
  - Store the new CTS.
  - Pass `cts.Token` to `executeAsync` instead of `CancellationToken.None`.
  - Treat `OperationCanceledException` from that token as a normal cancellation, not an error report.
  - In `finally`, clear and dispose the active CTS if it is still the current one.
- Add a public `Cancel()` method that cancels the active CTS. This keeps cancellation available without changing the existing command constructor API or all view-model call sites.

Testing:

- Add a focused test for `AsyncRelayCommand`:
  - Execute a command whose delegate captures the token and waits on a `TaskCompletionSource`.
  - Assert the captured token `CanBeCanceled` is true.
  - Call `Cancel()`.
  - Assert the captured token is cancellation requested.
  - Complete the task to avoid leaking async work.
- Add a second test if needed:
  - Delegate throws `OperationCanceledException` when its token is canceled.
  - Assert `reportException` is not invoked for expected cancellation.

### 4. Guard Discard Dialog Cancellation Close
In `WindowAppDialogService.ConfirmDiscardChangesAsync()`:

- Keep the cancellation registration.
- Change the posted callback to guard the close call:
  - `if (window.IsVisible) window.Close(false);`
- Optionally use `DispatcherPriority.Send` only if needed by tests; otherwise keep the current dispatcher post pattern.

Testing:

- This is hard to test without coupling to Avalonia modal-dialog behavior.
- If adding a headless test is straightforward, create a discard dialog through the service and exercise cancellation while visible.
- Otherwise rely on the code-level guard plus existing view-model discard-confirmation tests.

## Risk Notes

- The `AsyncRelayCommand` change is the only behavior-affecting item. It should preserve existing command construction and XAML binding while adding cancellable execution tokens.
- Re-entry cancellation should not affect most existing commands because view-model `IsBusy` gates already prevent normal repeated execution; it mostly fixes race/re-entry and testability.
- The window fixes follow existing patterns already used successfully in `WindowImageViewService`.
