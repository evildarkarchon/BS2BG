# Fix post-review findings: progress bar busy state

## Context

A local review of `codex/apply-openspec-change` vs `origin/main` surfaced three items:

1. ~~Path traversal bypass in `tools/release/package-release.ps1`~~ — **Retracted.** On closer reading, `$repoRootWithSeparator` already appends `Path.DirectorySeparatorChar`, so `StartsWith("J:\jBS2BG\", OrdinalIgnoreCase)` correctly rejects siblings like `J:\jBS2BG-evil\...`. The guard is sound. No change needed.
2. Progress bar in `MainWindow.axaml` only reflects `Templates.IsBusy`, hiding feedback during shell and Morphs operations. **In scope for this plan.**
3. Duplication of OpenSpec skills/workflows/commands across eight editor-specific directories. **Out of scope** — per the user, each agent-specific directory is required for the respective agent to discover its artifacts.

This plan covers only item 2.

## Goal

Make the `MainWindow` shell progress indicator visible whenever any of the three view models (`MainWindowViewModel`, `TemplatesViewModel`, `MorphsViewModel`) report `IsBusy = true`, so status-bar progress feedback matches the work actually in flight (open/save/export, template import, NPC import).

## Approach

Expose a single computed `IsAnyBusy` property on `MainWindowViewModel` and bind `ProgressBar.IsVisible` to it. This keeps the view dumb and matches the existing pattern of VM-driven state on this codebase (no new converters, no `MultiBinding`).

### Changes

#### 1. `src/BS2BG.App/ViewModels/MainWindowViewModel.cs`

- Add a public read-only computed property:
  ```csharp
  public bool IsAnyBusy => IsBusy || Templates.IsBusy || Morphs.IsBusy;
  ```
- In the `IsBusy` setter (currently at ~line 187), after `RaiseCommandStatesChanged()`, also call `this.RaisePropertyChanged(nameof(IsAnyBusy));`.
- In the constructor (after `Templates` and `Morphs` are assigned, around line 99), subscribe to child VM property changes:
  ```csharp
  Templates.PropertyChanged += OnChildBusyChanged;
  Morphs.PropertyChanged += OnChildBusyChanged;
  ```
  Where `OnChildBusyChanged` raises `IsAnyBusy` whenever the child's `IsBusy` property fires:
  ```csharp
  private void OnChildBusyChanged(object? sender, PropertyChangedEventArgs args)
  {
      if (args.PropertyName == nameof(TemplatesViewModel.IsBusy)
          || args.PropertyName == nameof(MorphsViewModel.IsBusy))
          this.RaisePropertyChanged(nameof(IsAnyBusy));
  }
  ```
- Verify both `TemplatesViewModel.IsBusy` and `MorphsViewModel.IsBusy` already raise `PropertyChanged` (they do — they use `RaiseAndSetIfChanged` via `ReactiveObject`).
- No `using` additions expected beyond `System.ComponentModel` if not already present (it is used indirectly via ReactiveUI but an explicit `using System.ComponentModel;` may be needed for `PropertyChangedEventArgs` — confirm at edit time).

#### 2. `src/BS2BG.App/Views/MainWindow.axaml`

- Change line 146:
  ```xml
  IsVisible="{Binding Templates.IsBusy}"
  ```
  to:
  ```xml
  IsVisible="{Binding IsAnyBusy}"
  ```

## Tests

Add a focused test to `tests/BS2BG.Tests/MainWindowViewModelTests.cs` (file already exists) covering three cases:

1. `IsAnyBusy` is false initially.
2. `IsAnyBusy` becomes true when `Templates.IsBusy` becomes true (simulate via the existing public mechanism — if `IsBusy` has a private setter, exercise it through an actual command such as `Templates.ImportPresetFilesAsync` driven by a stub picker; otherwise add a minimal internal helper in test scope). Prefer driving via public API.
3. `IsAnyBusy` raises `PropertyChanged` when it transitions.

If driving through public async commands is too invasive for a unit test, an acceptable alternative is asserting the subscription by toggling `MainWindowViewModel.IsBusy` via an open method already used during save/open flows (both `SaveProjectAsync` and `OpenProjectAsync` flip `IsBusy`); a stubbed `IFileDialogService` returning `null` will cause those flows to exit without setting `IsBusy`, so instead use a stub that returns a fake path and a `ProjectFileService` stub — but this expands scope. Simpler: expose an internal-visible `RaiseIsAnyBusyForTests` or rely on the existing `IsBusy` transitions during the save flow already covered by other tests in that file.

**Recommended test shape:** keep it minimal — subscribe to `PropertyChanged`, invoke a public async path that sets `IsBusy` (an existing test pattern in `MainWindowViewModelTests.cs` does this for save/open), and verify `IsAnyBusy` fired.
