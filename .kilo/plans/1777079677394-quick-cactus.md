# Plan: Round `SliderProfile` default percentages before casting to int

## Problem

`src/BS2BG.Core/Formatting/SliderProfile.cs:27-39` converts fractional
profile defaults to integer slider values with truncation:

```csharp
public int GetDefaultSmall(string sliderName) =>
    defaultsByName.TryGetValue(sliderName, out var value)
        ? (int)(value.ValueSmall * 100f)
        : 0;

public int GetDefaultBig(string sliderName) =>
    defaultsByName.TryGetValue(sliderName, out var value)
        ? (int)(value.ValueBig * 100f)
        : 0;
```

`SliderDefault.ValueSmall/ValueBig` are `float` values parsed from JSON
by `SliderProfileJsonService.ReadDefaults`
(`src/BS2BG.Core/Generation/SliderProfileJsonService.cs:27`) using
`GetSingle()`. Many percentages that look exact in JSON (e.g. 0.29,
0.58, 0.01) are not exactly representable as `float`. For these
inputs, `value * 100f` lands just below the intended integer
(e.g. `28.999996`), and `(int)` truncates toward zero, yielding 28
instead of 29, 57 instead of 58, 0 instead of 1, etc.

These defaults feed `SliderMathFormatter.ResolveSmall`/`ResolveBig`
(`src/BS2BG.Core/Formatting/SliderMathFormatter.cs:114-118`), which
drive BodySlide template lines and BoS JSON generation — so affected
profiles produce numerically wrong output.

Note: the existing fixture profiles under
`tests/fixtures/inputs/profiles/` only use exact values (0, 0.2, 1),
so current fixture-based tests do not exercise the defect.

## Goal

Convert fractional defaults to integer slider percentages using
rounding rather than truncation, so that profile values like 0.29,
0.58, 0.01 and similar survive the `float → int` conversion
correctly, while keeping current fixtures and behaviour unchanged for
exact values.

## Approach

Use `MathF.Round` with `MidpointRounding.AwayFromZero` in both
`GetDefaultSmall` and `GetDefaultBig`. Rationale:

- `AwayFromZero` matches the intuitive "percent rounding" a user
  expects (0.005 → 1, not 0 via banker's rounding) and avoids surprise
  differences from the default `ToEven` behaviour elsewhere.
- Staying with `float` (`MathF.Round`) keeps the signature and
  allocation profile unchanged and is sufficient: the issue is the
  final truncation step, not the scale.
- Negative defaults (if any) round symmetrically, so negative sliders
  are handled consistently.

A `decimal`-based conversion was considered but rejected: the inputs
are already `float` (precision lost at parse time), and the extra
conversion would not recover any information beyond what `MathF.Round`
provides.

### Code change (for reference only — no edits during plan mode)

```csharp
public int GetDefaultSmall(string sliderName) =>
    defaultsByName.TryGetValue(sliderName, out var value)
        ? (int)MathF.Round(value.ValueSmall * 100f, MidpointRounding.AwayFromZero)
        : 0;

public int GetDefaultBig(string sliderName) =>
    defaultsByName.TryGetValue(sliderName, out var value)
        ? (int)MathF.Round(value.ValueBig * 100f, MidpointRounding.AwayFromZero)
        : 0;
```

## Out of scope

- Changing the JSON schema or parsing (still `float` via
  `JsonElement.GetSingle()` in `SliderProfileJsonService`).
- Changing how `SliderMathFormatter` rounds other computations (e.g.
  percent interpolation or multiplier application) — only the
  profile-defaults-to-int conversion is addressed here.
- Adding new fixture JSONs with fractional defaults; unit tests cover
  the precision case directly without touching fixture expectations.

## Risks / considerations

- `MidpointRounding.AwayFromZero` vs default `ToEven`: only matters at
  exact halves (e.g. `0.005`). Defaults files are not expected to rely
  on midpoint behaviour, but `AwayFromZero` aligns with the "percent"
  mental model and is explicit, documented in the code, and covered
  by tests.
- No public API change; callers of `GetDefaultSmall`/`GetDefaultBig`
  continue to receive `int`.
