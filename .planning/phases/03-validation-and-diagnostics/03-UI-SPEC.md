# Phase 03 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | none — Avalonia desktop app, not React/Next/Vite; shadcn gate not applicable |
| Preset | not applicable |
| Component library | Avalonia 12 FluentTheme controls: `TabControl`, `Grid`, `Border`, `ListBox`, `TextBox`, `ComboBox`, `Button`, `Popup`, `GridSplitter`, `ProgressBar` |
| Icon library | none detected; use text labels and severity badges, not new icon dependency |
| Font | Inter through existing Avalonia Fonts.Inter package; generated output/snippets use Consolas monospace |

**Existing shell contract:** add a top-level `Diagnostics` tab to the existing `WorkspaceTabs` beside `Templates` and `Morphs`. Keep `MainWindow` as a plain Avalonia `Window`, use compiled bindings with `x:DataType` on every new `DataTemplate`, and follow the existing theme resources in `Themes/ThemeResources.axaml`.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | GridSplitter tracks, table row separators, compact text-to-control offsets |
| sm | 8px | Button groups, popup inner gaps, severity badge padding, row-to-detail spacing |
| md | 16px | Default Diagnostics panel padding, side-pane/content inset, empty-state padding, report section spacing |
| lg | 24px | Empty-state outer margin, major column gap before detail panes |
| xl | 32px | Reserved for wide summary-card groups if the Diagnostics tab uses cards |
| 2xl | 48px | Minimum pointer target for confirmation/action buttons; reserved for large centered empty-state vertical breathing room |
| 3xl | 64px | Reserved for page-level separation only; avoid inside dense desktop tables |

No spacing-token exceptions are allowed for Phase 3; use only `4, 8, 16, 24, 32, 48, 64`. Layout dimensions that are not spacing tokens: preview/filter popups may use 240px width when matching existing `MainWindow.axaml` patterns.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 12px | 400 | 1.4 |
| Heading | 16px | 600 | 1.2 |
| Display | 18px | 600 | 1.2 |

**Typography rules:** use exactly these weights: regular `400` and semibold `600`. Use 12px labels for severity chips, count badges, and secondary metadata; use 14px body for finding titles/details; use 16px semibold for panel headings; use 18px semibold only for tab-level headings and empty-state headings. Generated text snippets and file-output samples use `Consolas` at 14px so snippets remain on the declared 12/14/16/18 scale.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `BS2BGWindowBackgroundBrush` light `#F7F9FC`, dark `#11151A` | App background, Diagnostics tab background, report empty area |
| Secondary (30%) | Fluent chrome/resource surfaces (`SystemControlBackgroundChromeMediumBrush`) plus bordered `Border` surfaces | Finding groups, preview panes, popups, summary cards, row detail panels |
| Accent (10%) | `BS2BGFocusBrush` light `#005FB8`, dark `#7CC4FF`; active-filter resources for selected/filter state | Primary `Run Diagnostics`, selected finding indicator, active area filter, focus/navigation affordance, copy-report action |
| Destructive | `SystemControlErrorTextForegroundBrush` | Destructive actions only; Phase 3 has no destructive diagnostics actions. Use for true failed save/export error text and blocker count emphasis only when the finding blocks output readiness. |

Accent reserved for: `Run Diagnostics`, selected Diagnostics tab/finding state, active severity/area filters, focus outlines, `Copy Report`, and navigation affordances such as `Go to preset`/`Go to NPC`. Do not use accent for every button; routine secondary buttons remain Fluent default.

**Severity colors:** `Info` uses existing `BS2BGInfo*` brushes. `Caution` uses existing `BS2BGWarning*` brushes and the label `Caution`, never `Warning`. `Blocker` uses error/destructive foreground and a bordered neutral/error surface, with copy that explains blocked output readiness rather than blaming the user.

## Phase 3 Interaction Contract

### Diagnostics panel/tab

- Add `Diagnostics` as a first-class workspace tab, not a modal-only report and not a text-only dump.
- Default tab content is a two-pane or three-pane layout: summary/filter rail, grouped finding list, and selected-finding detail/drilldown. Use `GridSplitter` when panes are resizable.
- The summary/filter rail with blocker/caution counts is the first visual anchor; selected finding detail is secondary.
- Findings are read-only. Phase 3 may navigate/select related presets, targets, NPCs, or output areas, but must not auto-fix project data.
- Provide `Run Diagnostics` and `Copy Report` as visible actions in the tab header. `Run Diagnostics` refreshes the report; `Copy Report` copies a plain-text report with grouped findings and outcome ledgers.
- Group findings by: `Project`, `Profiles`, `Templates`, `Morphs/NPCs`, `Import`, and `Export`. Every row still shows `Blocker`, `Caution`, or `Info`.
- Severity labels are exactly `Blocker`, `Caution`, and `Info`. Avoid generic `Error`/`Warning` labels in the report UI.
- Profile diagnostics appear only inside the Diagnostics tab/report. Do not add ambient profile banners or warning copy to the normal Templates workflow.

### Profile diagnostics drilldown

- Whole-project profile diagnostics are the default. Selected-preset filtering is an optional drilldown/filter, not the only diagnostic scope.
- Summary must show coverage counts, unknown slider counts, injected defaults, multiplier count, inversion count, neutral fallback state, and affected preset count.
- Slider-level details are expandable/collapsible rows or a selected-detail pane. They must remain read-only and may reuse the existing slider-preview mental model from the SetSliders inspector.
- Unbundled saved profiles are `Info`, not `Caution` or `Blocker`: show saved profile name, calculation fallback profile, affected presets, and current fallback behavior.
- Do not implement likely profile mismatch scoring, slider-name mismatch heuristics, or Fallout 4 experimental language in Phase 3.

### NPC import preview

- Add an optional preview path. Existing direct import may remain, but preview must parse into a temporary result and must not mutate NPC database/project state until `Import Previewed NPCs` is invoked.
- Preview shows a summary plus row-level table for parsed rows, invalid lines, within-file duplicates, existing database/project duplicates where known, fallback-decoded files/rows, and rows that would be added.
- Keep import effects and assignment effects visually distinct. Import preview copy must say importing adds NPCs to the database only; assignment-changing commands get separate before/after counts.
- Fallback charset decoding is a per-file `Caution` with the encoding name from parser results.

### Export preview and failure ledger

- Export preview shows exact target paths, create-vs-overwrite state, batch/partial-output risk, and concise snippets before risky writes.
- Snippet default: first 3 non-empty lines per generated file/category plus total line/count summary. Full content preview is optional and must use generated output from existing Core services, not a separate formatter.
- Routine create-new exports do not require confirmation. Existing target files and multi-file partial-output risk require confirmation with the overwrite copy above.
- Save flow does not gain preview friction. Improve only failure messaging and outcome reporting.
- Failure/result ledger lists each file as `Written`, `Restored`, `Skipped`, `Left untouched`, or `Incomplete/unknown`, plus original exception and rollback/incomplete state when known.

### Accessibility and keyboard contract

- Every new actionable control must set `automation:AutomationProperties.Name` with a user-facing label.
- Preserve existing keyboard conventions and add no conflicting shortcuts. If adding a diagnostics shortcut, use `Ctrl+D` only if no existing command uses it and expose it in menu/automation text.
- Findings list must be keyboard navigable. Enter or an explicit `Go to...` button may navigate to affected data when available.
- Copy/report and preview/commit buttons must remain reachable through tab order after the shell search, theme selector, and command palette button.
- Do not rely on color alone: severity rows must include text labels and counts.
