## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | GridSplitter tracks, row dividers, compact field-label gaps |
| sm | 8px | Button groups, validation row gaps, profile-source badge padding, popup/dialog inner gaps |
| md | 16px | Default Profiles tab margin, panel padding, editor section spacing, dialog padding |
| lg | 24px | Profile list/detail separation, empty-state outer margin, conflict option spacing |
| xl | 32px | Reserved for wide summary-card groups or blank-profile onboarding blocks |
| 2xl | 48px | Minimum pointer target height for primary dialog actions and destructive confirmations |
| 3xl | 64px | Reserved for page-level vertical breathing room only; avoid inside dense editor tables |

No spacing-token exceptions are allowed for Phase 4; use only `4, 8, 16, 24, 32, 48, 64`. Layout dimensions that are not spacing tokens may match existing `MainWindow.axaml` patterns: 280px side rails, 320px inspector/detail panes, 240px popups, and 4px splitter tracks.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 12px | 400 | 1.4 |
| Heading | 16px | 600 | 1.2 |
| Display | 18px | 600 | 1.2 |

**Typography rules:** use exactly these weights: regular `400` and semibold `600`. Use 12px labels for source badges (`Bundled`, `Custom`, `Embedded`, `Missing`), table names, validation severity, and secondary metadata. Use 14px body for field labels, validation text, and profile descriptions. Use 16px semibold for panel headings. Use 18px semibold only for the Profiles tab heading, empty-state heading, and conflict-dialog title. JSON snippets use `Consolas` at 14px so snippets remain on the declared scale.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `BS2BGWindowBackgroundBrush` light `#F7F9FC`, dark `#11151A` | App background, Profiles tab background, editor empty area |
| Secondary (30%) | Fluent chrome/resource surfaces (`SystemControlBackgroundChromeMediumBrush`) plus bordered `Border` surfaces | Profile list, profile detail/editor cards, validation panels, conflict dialogs, JSON preview panes |
| Accent (10%) | `BS2BGFocusBrush` light `#005FB8`, dark `#7CC4FF`; active-filter resources for selected/filter state | Primary profile actions, selected profile row, active validation filter, focus/navigation affordance, exact-match recovery action |
| Destructive | `SystemControlErrorTextForegroundBrush` | Destructive actions only: delete local custom profile, replace local profile with embedded/imported data, discard unsaved edits, validation blockers |

Accent reserved for: `Import Profile`, `Save Profile`, `Copy Bundled Profile`, selected `Profiles` tab/profile row, active source or validation filters, focus outlines, and recovery CTAs (`Import Matching Profile`, `Remap Profile`). Do not use accent for every button; routine secondary buttons remain Fluent default.

**Status colors:** use existing `BS2BGInfo*` brushes for neutral fallback/recovery information and blank-profile guidance. Use existing `BS2BGWarning*` brushes for `Caution` rows such as extreme finite numeric values. Use destructive/error foreground only for strict validation blockers and explicitly destructive confirmations. Do not label unresolved custom profiles as `Error` or `Warning` in normal generation flows; keep Phase 1 neutral fallback tone.

## Phase 4 Interaction Contract

### Profiles workspace

- Add `Profiles` as a first-class workspace tab. Do not hide custom profile management behind file dialogs only.
- Default layout is a resizable two-pane `Grid`: left profile list/source rail, 4px `GridSplitter`, right profile detail/editor. Use a second vertical or horizontal splitter only if validation diagnostics and editor table need separate sizeable panes.
- Left rail groups profiles by source in this order: `Bundled`, `Custom`, `Embedded in project`, `Missing references`. Each row shows display name, source badge, and an optional affected-preset count.
- Bundled profile rows are selectable and inspectable but read-only. Their only authoring action is `Copy as Custom Profile`.
- Custom profile rows expose `Edit`, `Validate Profile`, `Export Profile JSON`, and `Delete Custom Profile`.
- Embedded project profiles are shown when present. If not installed locally, expose `Import as Custom Profile` or `Keep Embedded Only` according to the load/recovery state.
- Missing references are shown as neutral recovery items, not blockers. They link to affected presets and share copy with the Diagnostics tab.
- A secondary `Manage Profiles` button may appear beside the existing Templates profile selector, but it must navigate/open the same `Profiles` workspace state rather than duplicating a separate manager.

### Profile editor

- Editor sections are ordered: `Metadata`, `Defaults`, `Multipliers`, `Inverted Sliders`, `Validation`.
- Metadata fields include display name and game/style metadata. Display name is the identity users see and must be the only recovery identity; filenames are storage details.
- Slider tables use dense editable rows with columns: slider name, value/toggle, table, validation state, and row actions. Defaults and multipliers use numeric text boxes; inverted entries use a checkbox/toggle row or boolean text value consistent with the Core model.
- Table rows support add, remove, and edit. Removing rows from a custom profile is allowed but requires validation before save. Removing rows from bundled profiles is not available.
- `Save Profile` is disabled until the current editor state has no validation blockers. `Validate Profile` remains enabled when there are unsaved edits.
- Blank custom profiles are valid when well-formed and uniquely named. Show the blank-profile info callout; do not block creation solely because tables are empty.
- Broad finite numeric values are accepted. Do not clamp defaults to 0-1 or multipliers to positive values. Show `Caution` only for extreme values when implemented; never turn unusual finite values into blockers.
- Validation results are visible in the editor and summarized at the top. Blockers use text labels and row anchors; color alone is insufficient.

### Import, copy, export, and save flows

- `Import Profile` opens a JSON picker through the existing Avalonia StorageProvider service boundary. It validates before catalog inclusion and local-store writes.
- Imported profile identity comes from internal display-name metadata. The filename is never used to resolve a project reference.
- If imported data has a duplicate display name across bundled/custom profiles, reject the import unless the flow is an explicit conflict-resolution replace/rename decision.
- `Copy as Custom Profile` from a bundled profile pre-populates editor fields and requires the user to choose a non-conflicting display name before saving.
- `Create Blank Profile` opens the editor with empty `Defaults`, `Multipliers`, and `Inverted` tables plus the blank-profile info callout.
- `Export Profile JSON` is available only for selected custom or embedded profiles. It does not expand into Phase 5 portable bundles.
- Local profile save failures show actionable file-access copy and keep the editor open with unsaved changes intact.

### Missing custom profile recovery

- Project open remains non-blocking when a custom profile is missing. The user must land in the app with the project loaded and a visible neutral diagnostic/recovery state.
- Missing-profile recovery appears in both `Diagnostics` and `Profiles`. `Diagnostics` remains read-only by default; recovery actions are explicit buttons and must not run automatically when a finding is selected.
- Recovery actions are exactly: `Import Matching Profile`, `Remap to Installed Profile`, and `Keep Unresolved for Now`.
- `Import Matching Profile` resolves only by internal display-name exact match, case-insensitive. If the imported profile has a different internal name, show the recovery exact-match note and leave the reference unresolved.
- `Remap to Installed Profile` must clearly state that it changes the project preset profile reference and should participate in existing dirty/undo behavior.
- `Keep Unresolved for Now` keeps the saved reference and visible fallback calculation behavior. Do not silence the fallback information after this choice; lower friction, not visibility.

### Embedded profile conflict dialog

- When opening a project with embedded profile data that conflicts with a local custom profile of the same display name, show a blocking decision dialog before installing or replacing local data.
- The dialog offers: `Use Project Copy`, `Replace Local Profile`, `Rename Project Copy`, and `Keep Local Profile`.
- The dialog body must state that BS2BG will not silently choose either source. Show source metadata and a concise difference summary when available.
- `Replace Local Profile` is destructive and requires the destructive confirmation copy above or a two-step primary button treatment inside the conflict dialog.
- `Rename Project Copy` requires a unique display name before the dialog can complete.
- `Keep Local Profile` opens the project with local data active and preserves embedded data in the project model only as needed for save/share semantics; do not silently delete embedded data during load.

### Project embedding and sharing copy

- Saving a project embeds only custom profiles referenced by project presets. Do not show or include unrelated local custom profiles in sharing summaries.
- Save/share copy must explicitly say legacy fields remain present: existing `Profile` and `isUUNP` compatibility is preserved.
- Do not introduce Phase 5 portable bundle language. Phase 4 copy is limited to project-embedded custom profiles and selected-profile JSON export.

### Accessibility and keyboard contract

- Every new actionable control must set `automation:AutomationProperties.Name` with a user-facing label.
- Preserve existing keyboard conventions and add no conflicting shortcuts. If a profile-management shortcut is added, use the command palette first rather than adding an unverified global keybinding.
- Profile list, editor rows, validation findings, and conflict options must be keyboard navigable. Enter activates the selected explicit action only when focus is on a button or command row.
- Do not rely on color alone: profile source, validation severity, missing state, and destructive actions must have text labels.
- Dialog primary/secondary order must keep the safer action available: destructive primary buttons require explicit destructive wording, and cancellation/keep actions remain visible.
