<interfaces>
Formal algorithm spec required by review: strategy scope is an explicit eligible NPC list supplied by callers. CLI/core default is all `project.MorphedNpcs`; GUI Plan 05 will deliberately pass all rows for cross-machine determinism and show the count before apply. `ComputeEligibility(ProjectModel, AssignmentStrategyDefinition, IReadOnlyList&lt;Npc&gt; eligibleRows)` is the single non-mutating source used by both `Apply` and `ProjectValidationService`; duplicate eligibility logic is forbidden.

Weighted semantics: for each NPC, select matching rules in stable order by rule Name OrdinalIgnoreCase, then original rule order as tie-breaker. A matching rule is one whose RaceFilters is empty or contains `Npc.Race` OrdinalIgnoreCase and whose PresetNames resolve to existing project presets. Draw one integer from the provider in `[0,totalWeightUnits)` where totalWeightUnits is the sum of positive integer weight units after converting Weight to fixed two-decimal units (`Math.Round(weight * 100, MidpointRounding.AwayFromZero)`). Pick the first cumulative rule exceeding the draw, then pick a preset from that rule's eligible PresetNames using the same provider seam. All-zero/invalid weights were rejected by Plan 03.

GroupsBuckets semantics: bucket membership is rule-predicate based, not game lookup. A rule matches an NPC when `BucketName` is non-empty and RaceFilters contains `Npc.Race` OrdinalIgnoreCase; matching rules are ordered by BucketName then rule Name OrdinalIgnoreCase. Eligibility is the union of matching rules' PresetNames in project preset order. If no bucket rule matches or all preset names are unknown, that NPC is blocked per D-15.
