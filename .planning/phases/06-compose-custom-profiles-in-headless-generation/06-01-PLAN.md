<objective>
Close the Phase 6 AUTO-01 gap by making CLI/headless generation compose project custom profiles into the generation catalog before validation and output writing.

Purpose: `HeadlessGenerationService` currently receives a bundled-only constructor catalog and passes it directly to validation, `templates.ini`, and BoS JSON generation. Projects that already embed custom profile definitions can therefore silently fall back to bundled profile semantics in CLI generate, while portable bundles already use request-scoped profile composition. This reviewed plan makes the catalog rule shared Core behavior, locks the CLI scope to embedded project profiles, and proves the CLI output bytes match the embedded profile.
Output: A shared Core catalog composer, headless generation wiring, portable bundle reuse of the same composer, focused composer edge-case tests, and CLI/bundle regression tests that compare actual output bytes rather than only success/file existence.
</objective>

private TemplateProfileCatalog BuildRequestProfileCatalog(ProjectModel project, ProjectSaveContext? saveContext)
{
    var entries = profileCatalog.Entries
        .Where(entry => entry.SourceKind == ProfileSourceKind.Bundled)
        .Concat(ResolveBundleProfileSet(project, saveContext).Select(profile => new ProfileCatalogEntry(
            profile.Name,
            new TemplateProfile(profile.Name, profile.SliderProfile),
            profile.SourceKind,
            profile.FilePath,
            false)));

    return new TemplateProfileCatalog(entries);
}
